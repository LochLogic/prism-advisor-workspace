## Layer 1 — Experience & Design

**Strengths (verified across views/states/breakpoints/themes this session):** premium serif/navy/gold system; full dark-mode parity; demo opens on the lifecycle-roadmap wedge; inline interactive performance view; mobile overflow fixed; roster→cards; icon-only mobile switcher; live sparklines; every icon button labelled; WCAG-AA contrast after the recent fix; notification/alert deep-linking; 1-click demo with no signup wall.

- 🟠 **No error/empty fallback for crashes.** A thrown render error → blank page (see Frontend). For a pilot, this is the highest-impact CX risk.
- 🟡 **Admin view unverified live.** Demo is advisor-role; the firm-admin UI (Revenue/audit) couldn't be exercised here — needs one live admin login to confirm it renders (it's a pure refactor, low risk).
- 🟢 **New-advisor / new-client empty states** are good (welcome hero + "load sample household"; blank-slate nudge) — keep.
- 🟢 Login/signup pages are functional but plainer than the app — fine.

**Click-pathing:** golden paths are tight (open client → roadmap = 2 clicks; notification → context = 1; edit numbers = 1 from the modal after the Sprint-3 fix). No dead-ends found.

## Layer 2 — Engineering

**Frontend**
- 🟠 **No error boundaries anywhere** (only react-dom internals). One component throw crashes the SPA. Add `<ErrorBoundary>` around `AppInner` with a friendly fallback + a "reload" affordance.
- ✅ **XSS-safe:** both `dangerouslySetInnerHTML` paths (alert body, phase rationale) run through `sanitizeHtml` (escape-all + a tiny no-attribute tag whitelist) — safe even from a malicious firm-admin override.
- ✅ Refactor verified: `advisor-dashboard.jsx` split (2,665→955) into `advisor-modal.jsx` + `firm-admin.jsx`; client/advisor/modal all render with zero console errors.
- 🟢 Hook hygiene looked clean in spot-checks; a full hook-dependency audit is a follow-up (not a blocker).

**Backend**
- ✅ Edge Functions reviewed, hardened (CORS→prismaw.com, Plaid ownership check, constant-time cron compare) **and deployed** (verified live: `health` ok, CORS header = `https://prismaw.com`).
- 🟠 **6 `SECURITY DEFINER` functions don't pin `search_path`:** `px_current_advisor_id/_client_id/_firm_id`, `px_is_firm_admin`, `px_sign_acknowledgement`, `update_client_last_meeting`. (`px_provision_firm` and `rls_auto_enable` correctly set it — apply the same everywhere.) Fix: `ALTER FUNCTION … SET search_path = public, pg_temp;`
- ✅ Webhook signature-verified; invoices idempotent (unique constraint); ack signing via `SECURITY DEFINER` RPC so clients can't tamper with advisor fields.

**System Architecture**
- ✅ **Multi-tenant RLS is the standout.** Live-verified: **all 24 public tables have RLS enabled**; policy logic is correct and consistent (`firm→advisor→client` via `px_current_*` helpers); `aggregation_items` is RLS-on/0-policies = deny-all (correct for a service-role-only table).
- ✅ Trust boundaries clean: enforcement at the DB (RLS), secrets server-side only, demo-mode can't reach real data.
- 🟡 The no-module/window-globals model works and is now better-factored, but relies on load-order discipline (documented) — fragile for new contributors.

## Layer 3 — Performance

- 🟡 **Payload:** `dist/bundle.js` 247KB + vendor (react-dom 131KB, supabase-js 200KB, react 11KB) ≈ **590KB minified** (~160KB gzipped est.). Acceptable for an app; **supabase-js is the single heaviest dep.** Marketing/legal pages correctly load *only* supabase-client (no bundle). Cache-busting is content-hashed.
- 🟡 **`crm_tasks.assigned_to` is unindexed** — the task query filters `advisor_id OR assigned_to`; the assignee side can seq-scan. Add an index.
- 🟡 **Book-wide `balance_history` read** (Book-AUM sparkline) pulls the whole book's history; fine now, heavy at hundreds of clients × daily rows. Bound it (sample/monthly) later.
- ✅ Indexes otherwise solid (clients by advisor/firm/auth; balance_history; invoices; acks; audit; flagged_questions).
- **Scalability ceilings, in order:** Supabase free→Pro (do first); `balance_history` growth; roster DOM at 500+ clients (virtualization deferred — pagination bounds it for now); per-account Plaid cost vs flat pricing (unit-economics watch).

## Layer 4 — Safeguards

**Information Security** — strong overall.
- ✅ RLS everywhere + correct policies (see Architecture); `npm audit`: **0 vulnerabilities** (minimal deps; React/supabase vendored); CSP **live-verified hardened** (`script-src 'self'` + sha256 hashes, no `unsafe-inline`), HSTS + security headers; MFA/AAL2, PKCE, fail-closed roles.
- 🟠 `search_path` on the 6 SECURITY DEFINER functions (above).
- 🟠 **Plaid `access_token` stored plaintext** in `aggregation_items` — latent (Plaid is sandbox/parked); move to Vault before Plaid production.

**QA & Testing** — the weakest layer.
- ✅ **Unit:** 39 calc-core tests (the financial math).
- ◐ **Integration:** RLS harness exists but **not wired** (DATABASE_URL unset → CI skips). No automated coverage of Edge Functions (webhook→subscription, invoice run, ack RPC) or the auth/provision flow.
- 🟠 **No integration/E2E coverage of the critical flows** — these are exactly where a pilot breaks. At minimum: wire RLS→CI, and add a smoke test for signup→provision.
- **UAT:** none defined — script provided below (reuse it as your test-user onboarding guide).

## Layer 5 — Operations

**DevOps & CI/CD**
- ✅ CI: `npm ci` → build → check → calc tests (+ RLS job that skips until the secret).
- 🟡 **CI and deploy are decoupled** — Cloudflare auto-deploys on push to `main` regardless of CI result, so a red build still ships. (Mitigate: branch protection requiring CI, or a deploy gate.)
- 🟡 **Migrations are hand-run** and **Edge Functions are deployed manually via CLI** — both are drift risks with no automated reconciliation.
- 🟢 No automated rollback (Cloudflare keeps deploy history → manual rollback available).

**Monitoring & Telemetry**
- 🟠 **No error monitoring** — client crashes and Edge Function errors are invisible. Combined with the missing error boundary, a pilot bug = silent blank screen. This is the #1 operational gap for test users.
- 🟡 **No product analytics** — signup/activation funnel is invisible; you won't see where pilot users drop off.
- ✅ `audit_log` is solid internal telemetry; `/health` exists (nothing watches it); SEO health/digest workflows run weekly.

## UAT script (persona acceptance — also your test-user guide)
**Advisor:** sign up → name firm (provision) → add a client (with starting numbers) → open client → build roadmap (check milestones) → open a calculator → request an acknowledgement → generate/print a report. *Pass = each step completes with correct data and no console error.*
**Client:** log into portal → see roadmap → flag a question → review & e-sign the acknowledgement → view inline performance → update "your numbers". *Pass = sign records an immutable row; numbers flow into the plan.*
**Admin:** open Firm Admin → create a fee schedule → assign it to clients inline → run billing → approve an invoice → view the audit trail. *Pass = invoice generated + audit entries present.*
Run all three on desktop **and** mobile, light **and** dark.
